# Phase 0 Research: App Internationalization

**Feature**: 015-app-i18n-language-switcher | **Date**: 2026-05-22

This document records the decisions that resolve the open questions from the
Technical Context. The reference material the user supplied (`docs/intl.md`, the
Intlayer TanStack Start guide) and the i18next plugin catalogue were both reviewed.

## D1 — i18n library: react-i18next

**Decision**: Use `i18next` + `react-i18next` for the frontend. Supporting packages:
`i18next-browser-languageDetector` (runtime) and `i18next-cli` (devDependency). No
runtime backend/caching plugins.

**Rationale**:

- The feature explicitly forbids a URL locale (FR-006) and drives language from a
  settings switch. Intlayer's TanStack Start integration (per `docs/intl.md`) is built
  around locale-aware routing — `{-$locale}` route segments, `LocalizedLink`,
  `useLocalizedNavigate`, a locale layout, hreflang sitemap. In `no-prefix` mode all of
  that is discarded, leaving Intlayer used against the grain of its own documentation.
- react-i18next has no routing concept at all — the no-URL-locale requirement *is* its
  default behaviour. It is the long-established standard, framework-agnostic, and works
  with TanStack Start's SSR with well-understood patterns.
- Catalogs are **static assets**, not per-user data. Only the *active* locale is
  dynamic. So catalogs are bundled by Vite (English eager, Russian as a lazy
  `import()` chunk). Runtime backends (`i18next-http-backend`, `-fs-backend`,
  `-chained-backend`, `-localstorage-backend`) solve CDN/remote-catalog problems we do
  not have — rejected as YAGNI (Constitution III).
- `i18next-icu` is **not** needed: Russian's plural categories (one/few/many/other) are
  handled by i18next's native pluralization, which is backed by `Intl.PluralRules`.
- `i18next-cli` is included as a devDependency for two real wins: generating TypeScript
  types from `en.json` (so `t()` keys are typed and autocompleted) and linting for
  missing/unused keys — a CI gate that satisfies Constitution II for translation
  completeness. The generated type file `src/i18n/resources.d.ts` is **committed** to
  version control; CI runs `pnpm i18n:types && git diff --exit-code` to fail the build
  on drift, plus a `pnpm typecheck` step (added with this feature — currently absent
  from CI) so the typed keys are actually enforced. See tasks.md T015/T044.

**Alternatives considered**:

- *Intlayer in `no-prefix` mode* — the library from the reference doc. Genuine
  strengths: autogenerated typed content and an auto-extractor for migrating the
  existing hardcoded strings. Rejected because it is newer/smaller-community and would
  run counter to its documented happy path; the typed-key benefit is matched well
  enough by `i18next-cli` type generation.
- *FormatJS / react-intl, LinguiJS* — viable but offer no advantage here over the
  more widely used react-i18next; ICU tooling is unnecessary for a 2-locale app.
- *`vite-plugin-i18next-loader` / `i18next-hmr`* — optional conveniences (multi-file
  catalog compositing, hot-reload of translations). Not adopted now; plain JSON imports
  suffice. May be added later purely for DX without changing the architecture.

## D2 — Persistence: nullable `platform_users.locale`

**Decision**: Add a **nullable** `locale text` column to the control-plane
`platform_users` table, constrained by `CHECK (locale IN ('en','ru'))`. NULL means the
user has never explicitly chosen a language.

**Rationale**:

- FR-004 requires a per-account preference that follows the user across devices —
  account-level storage, not a cookie alone. `platform_users` is the right home: locale
  is a personal identity attribute, control-plane, no tenant scoping or RLS.
- FR-008 requires that when a signed-out visitor who picked a language signs in, an
  *existing* account preference wins, but if the account has *no* preference the
  signed-out choice is adopted. A `NOT NULL DEFAULT 'en'` column cannot distinguish
  "deliberately chose English" from "never chose" — so the column is nullable with no
  default. FR-014 (stored value no longer supported) is covered by treating an
  unrecognised value like NULL at read time.

**Alternatives considered**: `NOT NULL DEFAULT 'en'` — simpler, but loses the
"unset" state FR-008 needs. A separate `user_preferences` table — over-engineered for a
single scalar (Constitution III/VI: layer scope proportional to need).

## D3 — SSR no-flash via a `nv_locale` cookie

**Decision**: Maintain a cookie `nv_locale` that mirrors the *effective* locale.
TanStack Start's server render reads it to seed the i18next instance's `lng` and the
`<html lang>`/`dir` attributes, so the server's first paint already matches the user's
language.

**Rationale**:

- TanStack Start renders on the server (Nitro). Without a server-readable locale, the
  server would render English and the client would re-render — a visible flash,
  violating the spirit of FR-005/FR-011.
- A cookie is the only locale signal available to the server before any API call. The
  Go API writes `nv_locale` on `login`, `signup`, `accept-invitation`, and
  `PUT /api/platform/me`, always setting it to the effective locale. The client's
  language detector also caches to the same cookie. The cookie is a *mirror* for SSR;
  the DB column remains the source of truth for signed-in users.

**Alternatives considered**: localStorage — not readable by the server, so it cannot
prevent the flash. A URL locale — explicitly forbidden (FR-006).

## D4 — Effective-locale precedence

**Decision**: The effective locale is resolved in this order, first match wins:

1. **DB preference** — `platform_users.locale`, when the user is signed in and the
   value is non-NULL and supported.
2. **`nv_locale` cookie** — a previous explicit choice (signed-out visitor, or the SSR
   mirror).
3. **Browser** — `Accept-Language` header (server) / `navigator.languages` (client);
   first supported match.
4. **Default** — English.

**Rationale**: directly encodes Story 1 (account preference is authoritative for
signed-in users), Story 2 (browser detection for first-time/signed-out visitors, and
an earlier manual choice honoured over the browser), and FR-007/FR-008/FR-014. On the
client, `i18next-browser-languageDetector` is configured `order: ['cookie','navigator']`,
`lookupCookie: 'nv_locale'`, `caches: ['cookie']`; the DB value (from `GET /me`)
overrides the detector once the session is known.

**FR-008 adopt-on-sign-in**: `login`/`signup` read the incoming `nv_locale` cookie; if
the authenticated user's stored `locale` is NULL and the cookie carries a supported
locale, the API persists it as the account preference. An existing non-NULL preference
is never overwritten.

## D5 — Switcher location: a platform-plane Account page

**Decision**: Add a new platform-plane route `/account` with an Account settings page
that hosts the language switcher. It is not tenant-scoped.

**Rationale**: locale is a *personal* preference (the spec is explicit: "user should be
able to switch"). The only settings screen today — `/t/$slug/settings` — is
*workspace*-scoped (`display_name`, `timezone` apply to the whole tenant). Putting a
per-user control there would be semantically wrong and ambiguous in a multi-workspace
account. A dedicated platform-plane Account page is the correct home and leaves room
for other personal settings later without forcing that now (Constitution III).

**Alternatives considered**: a quick switcher in the top bar only — fails the spec's
"in the settings" wording and SC-006 (users locate it *in settings*); a top-bar
shortcut may still be added later as a convenience. Reusing the workspace settings page
— rejected as above.

## Notes on scope

- **Out of scope** (confirmed against spec Assumptions): backend-generated emails,
  exported files, and the tenant-branded **public subscription pages** — those are
  subscriber-facing and their language is a subscriber concern, not the signed-in
  user's UI preference. This feature localizes the authenticated app interface plus its
  pre-authentication surface (login, signup, invitation acceptance).
- **Translation rollout** is per-namespace and incremental. Because `fallbackLng: 'en'`
  ships with the foundational setup, a namespace that is not yet fully translated to
  Russian still renders (English fallback) — Story 3's guarantee is therefore available
  from the first increment, not a later add-on.
