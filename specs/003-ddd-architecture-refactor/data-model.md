# Phase 1 Data Model: Domain Entities

This describes the **domain layer** after the refactor — the Go entities, their
invariants, constructors, and state transitions. It is not a database schema:
the PostgreSQL tables, columns, and the RLS model are unchanged from
`specs/002-tenancy-core/data-model.md`. What changes is that the rules currently
spread across free functions and inlined SQL move onto these types.

## Conventions

Every aggregate follows the same shape (constitution Principle VI, PATTERNS.md #3):

- **Unexported fields**, exposed through getter methods. No public setters.
- **One validating constructor** `NewX(...) (*X, error)` — the only way to build
  a *new* entity; it enforces all invariants so an invalid instance cannot exist.
- **One hydration function** `HydrateX(...) *X` — used solely by repository
  adapters to reconstruct an entity from a persisted row. Documented inline as
  "persistence only — not a constructor"; it does not re-run validation.
- **Behavior methods** describe domain actions and guard transitions; they return
  typed `apperr` errors on invariant violations.

## auth context

### User

A platform identity. Never carries the password hash outside the persistence
boundary.

| Field | Type | Notes |
|---|---|---|
| id | string (UUID) | assigned by the database on insert |
| email | Email (value object) | normalized, case-insensitive |
| name | string | non-empty, trimmed |

- **Value object — Email**: trims surrounding whitespace; must match the
  plausible-address shape (`^[^@\s]+@[^@\s]+\.[^@\s]+$`). Construction of an
  invalid Email fails with `apperr` category `IncorrectInput`.
- **Value object — Password** (transient, never stored on User): plaintext must
  be 8–72 bytes. Validated at construction; the bytes are handed to the password
  adapter for hashing and not retained on any entity.
- **Constructor** `NewUser(email, name)` — validates Email and non-empty name.
- **Hydration** `HydrateUser(id, email, name)` — repository load.
- **Invariants**: email shape; name non-empty.

### Session

A login session bound to a user.

| Field | Type | Notes |
|---|---|---|
| id | string (UUID) | database-assigned |
| userID | string (UUID) | owning user |
| expiresAt | time.Time | issue time + TTL |
| revokedAt | *time.Time | nil while live |

- **Constructor** `NewSession(userID, ttl)` — sets `expiresAt = now + ttl`.
- **Hydration** `HydrateSession(id, userID, expiresAt, revokedAt)`.
- **Behavior**: `IsLive(now)` → not revoked and not expired; `Revoke(now)`.
- **Invariants**: a Session is usable only while `IsLive` holds. The raw token
  is **not** a field — it is generated by `internal/token`, returned once by the
  issuing command, and only its hash is persisted (FR-015).

## tenant context

### Tenant (Workspace)

An isolated workspace; the root of the tenant plane.

| Field | Type | Notes |
|---|---|---|
| id | string (UUID) | database-assigned |
| slug | Slug (value object) | unique, URL path segment |
| name | string | non-empty, trimmed |
| status | TenantStatus | `active` (only value this phase) |

- **Value object — Slug**: 3–63 chars, lowercase letters/digits/interior
  hyphens, starts and ends alphanumeric, not in the reserved set (`api`, `t`,
  `admin`, `login`, `signup`, `invite`, `static`, `assets`, `healthz`). A
  `DeriveSlug(name)` helper builds a candidate Slug from a workspace name.
  Invalid slug → `apperr` category `IncorrectInput`.
- **Constructor** `NewTenant(name, slug)` — validates name and Slug; derives the
  slug from the name when the supplied slug is empty.
- **Hydration** `HydrateTenant(id, slug, name, status)`.

### Membership

The association of a user with a tenant in a role.

| Field | Type | Notes |
|---|---|---|
| userID | string (UUID) | platform user |
| tenantID | string (UUID) | tenant |
| role | Role | `owner` or `admin` this phase |

- **Constructor** `NewMembership(userID, tenantID, role)` — validates the role.
- **Hydration** `HydrateMembership(...)`.
- The tenant creator gets `owner`; an accepted invitee gets `admin`.

### Invitation

A pending, expiring grant of tenant membership.

| Field | Type | Notes |
|---|---|---|
| id | string (UUID) | database-assigned |
| tenantID | string (UUID) | inviting tenant |
| email | Email | invitee address |
| status | InvitationStatus | `pending` → `accepted` / `revoked` |
| invitedBy | string (UUID) | inviting user |
| createdAt | time.Time | |
| expiresAt | time.Time | created + invite TTL |

- **Constructor** `NewInvitation(tenantID, email, invitedBy, ttl)` — validates
  Email, sets `status = pending`, `expiresAt = now + ttl`.
- **Hydration** `HydrateInvitation(...)`.
- **Behavior / state machine**:
  - `IsAcceptable(now)` → `status == pending` and not expired.
  - `Accept(now)` → requires `IsAcceptable`; sets `status = accepted`. Otherwise
    returns `apperr` `invitation_not_found` (the opaque error — an expired,
    revoked, or already-accepted invitation is indistinguishable, FR-015).
  - `Revoke()` → requires `status == pending`; sets `status = revoked`.
- The raw invitation token is not a field: generated by `internal/token`,
  returned once by `InviteTeammate`, persisted only as a hash.

```text
            NewInvitation
                 │
                 ▼
            ┌──────────┐  Accept(now)   ┌───────────┐
            │ pending  │ ─────────────▶ │ accepted  │
            └──────────┘                └───────────┘
              │      │
     Revoke() │      │ expiresAt passes (implicit — no row change;
              ▼      ▼   IsAcceptable returns false)
        ┌──────────┐ ┌──────────┐
        │ revoked  │ │ expired  │
        └──────────┘ └──────────┘
```

### TenantSettings

Per-tenant configuration — the first tenant-plane entity, reached only through
the RLS-bound transaction in the settings adapter.

| Field | Type | Notes |
|---|---|---|
| tenantID | string (UUID) | owning tenant |
| displayName | string | non-empty, trimmed |
| timezone | string | non-empty, trimmed |

- **Constructor** `NewTenantSettings(tenantID, displayName)` — initial row created
  with the tenant; `timezone` defaults at the database level.
- **Hydration** `HydrateTenantSettings(tenantID, displayName, timezone)`.
- **Behavior**: `Rename(displayName)`, `SetTimezone(tz)` — each rejects an empty
  value with `apperr` `IncorrectInput`.

## Repository interfaces (consumer-owned)

Declared in each context's `domain/repository.go`, implemented in `adapters/`.
Full signatures are in [contracts/repositories.md](./contracts/repositories.md).

| Interface | Aggregate | Key operations |
|---|---|---|
| `UserRepository` | User | create, get-by-id, lookup-by-email |
| `SessionRepository` | Session | issue, resolve-by-token-hash, revoke |
| `TenantRepository` | Tenant + Membership | create-workspace (tx), get-by-slug, get-by-id, list-memberships, get-role, add-membership, list-members |
| `InvitationRepository` | Invitation | create, get-pending-by-token-hash, list-pending, update-with-closure |
| `SettingsRepository` | TenantSettings | get (RLS-bound), update (RLS-bound) |
| `PasswordHasher` | — (auth app) | hash, verify — implemented by the bcrypt adapter |

## Read models

Query handlers return flat structs shaped for the HTTP response, **not** domain
entities (PATTERNS.md #4) — e.g. `MembershipView{ TenantID, Slug, Name, Status,
Role }`, `MemberView{ UserID, Email, Name, Role }`, `InvitationView{ ID, Email,
Status, CreatedAt, ExpiresAt }`. Their JSON shapes match the current responses
exactly (FR-001); they are defined next to the query handler that produces them.
